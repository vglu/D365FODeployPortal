# 🛡️ Двухуровневая валидация энвайронмента при деплойменте

## Цель

Гарантировать с **максимальной уверенностью**, что пакет D365FO будет применён на правильном энвайронменте. Если что-то пойдёт не так — обнаружить это как можно раньше.

---

## Архитектура валидации

### CHECK 1: PRE-DEPLOY валидация (перед началом деплоймента)

**Когда:** Сразу после `pac auth create`, **ДО** начала `pac package deploy`  
**Как:** Вызов `pac auth who` и парсинг вывода  
**Цель:** Убедиться что PAC CLI подключился к правильному энвайронменту  

**Результат:**
- ✅ Если правильный энв → продолжаем деплоймент
- ❌ Если неправильный энв → **Exception ДО начала деплоя** (пакет НЕ применён)

**Преимущества:**
- Ранний отказ — пакет не будет применён на неправильном энве
- Быстрая проверка (< 1 секунды)

**Код:**
```csharp
// src/DeployPortal/Services/Deployment/Validation/PreDeployAuthValidator.cs

onLog?.Invoke("[Pre-Deploy Validation] Verifying 'pac auth who' output...");

var expectedUrl = context.Environment.Url.ToLowerInvariant();
var expectedName = context.Environment.Name.ToLowerInvariant();
var whoOutputLower = context.PacAuthWhoOutput.ToLowerInvariant();

// Check 1: Match by URL (works for Service Principal auth)
var urlMatch = whoOutputLower.Contains(expectedUrl);

// Check 2: Match by Organization Friendly Name (works for interactive auth)
var friendlyNameMatch = 
    whoOutputLower.Contains($"organization friendly name: {expectedName}") ||
    whoOutputLower.Contains($"default organization: {expectedName}");

if (!urlMatch && !friendlyNameMatch)
{
    throw new InvalidOperationException(
        $"PAC CLI authenticated to WRONG environment!\n" +
        $"Expected: {context.Environment.Name} ({context.Environment.Url})\n" +
        $"But 'pac auth who' output does not contain expected URL or Organization Name.");
}

var matchType = urlMatch ? "URL" : "Organization Friendly Name";
onLog?.Invoke($"[Pre-Deploy Validation] ✓ Matched by {matchType}: {context.Environment.Name}");
```

**Почему две проверки?**
- **Service Principal auth** → выводит URL в `pac auth who`
- **Interactive auth (device code)** → выводит только "Organization Friendly Name"

### CHECK 2: POST-DEPLOY валидация (после завершения деплоймента)

**Когда:** После успешного выполнения `pac package deploy`  
**Как:** Парсинг лог файла, поиск строки `Deployment Target Organization Uri:`  
**Цель:** Финальное подтверждение что пакет применён на правильном энвайронменте  

**Результат:**
- ✅ Если правильный энв → деплоймент успешен
- ❌ Если неправильный энв → **Exception, деплоймент Failed** (пакет УЖЕ применён на неправильном энве!)

**Преимущества:**
- Финальное подтверждение из самого PAC CLI (из его логов)
- Обнаруживает случаи когда PAC CLI "передумал" в процессе деплоя

**Код:**
```csharp
// src/DeployPortal/Services/DeployService.cs, метод ValidateDeploymentLogAsync

var logContent = await File.ReadAllTextAsync(logFilePath);

// Find: "Deployment Target Organization Uri: https://cst-hfx-tst-07.crm.dynamics.com/..."
var uriLinePrefix = "Deployment Target Organization Uri:";
var uriLine = logContent
    .Split('\n')
    .FirstOrDefault(line => line.Contains(uriLinePrefix, StringComparison.OrdinalIgnoreCase));

var actualUri = ExtractUriFromLine(uriLine);

if (!actualUri.Contains(environment.Url, StringComparison.OrdinalIgnoreCase))
{
    var errorMsg = 
        $"❌ POST-DEPLOYMENT VALIDATION FAILED! ❌\n" +
        $"Package was deployed to WRONG environment!\n\n" +
        $"Expected: {environment.Url}\n" +
        $"Actual: {actualUri}";
    
    throw new InvalidOperationException(errorMsg);
}

onLog?.Invoke($"[Post-Deploy Validation] ✓ Confirmed: package was deployed to correct environment.");
```

## Поток выполнения

```
START
  ↓
1. Создать изолированную папку pac_auth_{deploymentId}_{guid}
  ↓
2. pac auth create (в изолированной папке)
  ↓
3. [CHECK 1 - PRE-DEPLOY]
   pac auth who → парсинг → проверка URL
   ↓
   Правильный энв?
   ├─ ДА → продолжаем
   └─ НЕТ → throw Exception (пакет НЕ применён) ❌
  ↓
4. pac package deploy
  ↓
5. [CHECK 2 - POST-DEPLOY]
   читаем лог файл → парсинг → проверка "Deployment Target Organization Uri"
   ↓
   Правильный энв?
   ├─ ДА → SUCCESS ✅
   └─ НЕТ → throw Exception, деплоймент Failed (пакет УЖЕ применён на неправильном энве!) ⚠️
  ↓
6. Удалить изолированную папку
  ↓
END
```

## Когда какая проверка сработает

| Сценарий | CHECK 1 (PRE-DEPLOY) | CHECK 2 (POST-DEPLOY) | Результат |
|----------|----------------------|------------------------|-----------|
| Всё хорошо | ✅ Пройден | ✅ Пройден | SUCCESS |
| PAC auth подключился к неправильному энву | ❌ Провален | Не выполняется | FAILED, пакет НЕ применён ✓ |
| PAC auth подключился правильно, но деплоймент пошёл на другой энв (маловероятно) | ✅ Пройден | ❌ Провален | FAILED, пакет применён на неправильном энве ⚠️ |
| Лог файл не найден или не содержит Uri | ✅ Пройден | ⚠️ Warning (не падаем) | SUCCESS (продолжаем, но логируем предупреждение) |

## Почему две проверки?

1. **CHECK 1 (PRE-DEPLOY)** — основная защита:
   - Ловит 99% случаев неправильного выбора энвайронмента
   - Прерывает деплоймент ДО применения пакета
   - Быстро работает

2. **CHECK 2 (POST-DEPLOY)** — финальное подтверждение:
   - Ловит маловероятные случаи когда PAC CLI "передумал" в процессе деплоя
   - Подтверждает что мы смотрим на те же данные что и PAC CLI (его собственный лог)
   - Даёт 100% уверенность для audit trail

**Вместе они дают максимальную защиту** от деплоймента на неправильный энвайронмент.
